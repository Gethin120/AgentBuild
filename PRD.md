# 结伴出行会合点规划产品 PRD

## Problem Statement

周末结伴出游时，同行人常常从不同地点出发，但目的地相同。现有地图与导航产品通常只能高效解决单人从 A 点到 B 点的问题，难以帮助两位出行者在多种限制条件下快速决定“在哪里会合最合适”。

对于一位驾车用户和一位乘坐公交地铁的朋友而言，真正需要决策的不是某条单一路线，而是一个兼顾双方体验的会合方案。用户通常会同时考虑多个约束，例如朋友公共交通耗时不能太久、司机绕路不能太多、双方等待时间不能太长、会合地点要足够好找、会合后继续前往共同目的地要顺路。

目前用户往往只能用多个地图 App 来回切换、手动尝试多个会合点、反复比较时间成本，决策过程繁琐，且难以兼顾公平性与效率。这导致出游前沟通成本高、决策慢、体验差，也容易因信息不对称产生分歧。

本产品要解决的问题是：帮助周末结伴出游的人，在双地出发、共同前往目的地的场景下，通过自然语言输入约束与偏好，自动生成可执行、可解释、可比较的会合方案，降低沟通成本并提升出发前决策效率。

## Solution

打造一个面向周末结伴出游场景的约束型会合点规划产品。用户用自然语言输入双方出发地、共同目的地以及偏好和限制条件，系统自动生成多个满足约束的会合方案，并用清晰易懂的方式解释每个方案的取舍。

产品默认面向典型的双人结伴场景：

- 一位用户驾车出发
- 一位朋友乘坐公交或地铁出发
- 双方汇合后共同前往同一目的地

系统围绕以下核心能力构建：

- 理解自然语言输入中的地址、角色、目的地、时间限制与偏好
- 自动生成候选会合点，如地铁站、商场、停车场、交通枢纽等
- 基于确定性规则引擎评估各候选点的可行性
- 综合乘客公共交通时长、司机绕路时长、双方等待时长和汇合后整体到达效率进行排序
- 输出 2 到 3 个推荐方案，并明确说明各方案的优先级与取舍

对用户而言，产品不是单纯返回一条路线，而是在帮他们做“会合决策”。系统需要让用户快速回答以下问题：

- 我们应该在哪里碰头
- 这个点为什么比别的点更合适
- 如果我更看重少等待或少绕路，应该选哪个方案
- 当前约束太严格时，应该放宽哪一项更合理

## User Stories

1. 作为周末自驾出游的人，我希望输入我和朋友的出发地以及共同目的地后，系统能自动推荐会合点，这样我不用手动在多个地图 App 之间反复比较。
2. 作为周末乘坐公共交通出游的人，我希望系统限制我的公交地铁通勤时间，这样我不会为了配合会合而承担过长换乘或通勤成本。
3. 作为驾车出发的人，我希望设置自己最多可接受的绕路时间，这样推荐方案不会严重偏离我的主路线。
4. 作为双方中的任意一方，我希望设置最大等待时间，这样会合体验更公平，不会出现一方长时间空等。
5. 作为计划出游的用户，我希望用自然语言描述需求，例如“朋友地铁别超过 100 分钟，我最多绕路 90 分钟”，这样输入成本更低。
6. 作为首次使用产品的用户，我希望系统自动识别出发地、目的地和角色关系，这样我不需要理解复杂表单。
7. 作为对地点不熟悉的用户，我希望系统自动生成可会合的候选点，这样我不用自己猜测哪里方便碰头。
8. 作为谨慎决策的用户，我希望一次看到多个可行方案，而不是只有一个答案，这样我可以在不同偏好之间做权衡。
9. 作为重视效率的用户，我希望系统告诉我每个方案的总耗时，这样我可以判断哪种方案更快到达目的地。
10. 作为重视公平的用户，我希望系统展示双方各自的出行成本，这样我能判断会合安排是否合理。
11. 作为驾车用户，我希望系统告诉我相比直接去目的地会多绕多少时间，这样我能直观看到接朋友的额外成本。
12. 作为公共交通出行的朋友，我希望系统告诉我会合前需要花多久以及是否需要复杂换乘，这样我能判断自己是否愿意接受该方案。
13. 作为结伴出游者，我希望系统告诉我谁会先到、谁需要等多久，这样我们可以提前协调出发时间。
14. 作为时间敏感的用户，我希望系统能在约束过严导致无解时给出放宽建议，这样我能快速得到可执行备选方案。
15. 作为经常出游的用户，我希望系统支持“优先少等待”“优先少绕路”“优先快到达”等偏好，这样结果更符合我的真实决策方式。
16. 作为注重可信度的用户，我希望产品解释为什么推荐这个会合点，例如它更顺路、双方更均衡或总体更快，这样我更愿意采纳系统建议。
17. 作为和朋友一起看方案的用户，我希望结果表达足够清晰，这样我们在聊天中就能快速达成一致。
18. 作为组织者，我希望可以将推荐的会合方案分享给同行人，这样不需要再次手动描述地点和理由。
19. 作为目的地较远的用户，我希望系统把“会合后继续前往目的地”的路程也纳入判断，这样推荐不会只优化会合前、不优化整体行程。
20. 作为跨城区出游的用户，我希望系统能处理不同城区甚至跨城的出发与目的地组合，这样周末长途出游也能使用。
21. 作为对地点可达性敏感的用户，我希望系统更偏向交通便利且好找的会合点，这样实际执行更顺畅。
22. 作为不愿折腾的用户，我希望系统默认优先地铁站、停车场、商场等易描述、易停靠、易上车的地点，这样线下会合更简单。
23. 作为临近出发的用户，我希望如果一方临时晚到或路线拥堵，系统能支持重新规划，这样我不用从头再算一遍。
24. 作为结果导向的用户，我希望系统能直接推荐“最优方案”“最省等待方案”“最省绕路方案”，这样我不必自己做复杂换算。
25. 作为产品运营或产品经理，我希望知道哪些请求成功返回了方案、哪些约束最常导致无解，这样可以持续优化候选点策略与默认配置。

## Implementation Decisions

- 产品聚焦在“周末结伴出游”的主场景，不尝试覆盖所有导航或调度问题。
- 第一阶段优先支持双人、双地出发、单一共同目的地的规划模式。
- 产品角色模型默认包含司机与公共交通乘客两类角色，以减少复杂度并贴合高频真实场景。
- 自然语言是首要输入方式，用户不需要先学习结构化表单；系统负责将自然语言转化为结构化约束。
- 会合点生成采用“自动候选点 + 规则筛选”的方式，以保证用户输入简单且系统结果可控。
- 候选点优先选择交通便利、现实中容易执行的地点类型，如地铁站、公交站、停车场、商场、交通枢纽。
- 规划结果必须由确定性规则与地图能力给出，模型只负责意图理解、策略协商和结果解释。
- 强约束不能被模型绕过，但应只来自用户明确表达或产品显式确认；系统不应在用户未说明时偷偷注入乘客通勤、司机绕路或等待上限。地理编码可信性仍属于系统必须守住的底线。
- 排序策略围绕四类核心指标展开：乘客会合前通勤成本、司机接人绕路成本、双方等待成本、会合后整体到达效率。
- 第一版结果输出至少包含三个层次：推荐方案、核心指标、推荐理由。
- 产品默认一次返回多个备选方案，至少包括均衡方案，并尽可能覆盖“最快到达”“最少等待”“最少绕路”等决策偏好。
- 当当前约束下无解时，产品不应只返回失败，而应尽量说明无解原因并提供合理的放宽建议。
- 系统应保留规划过程中的关键状态与评审结果，用于后续调试、策略优化和产品指标分析。
- 第一版可以以命令行或对话式形式承载，但产品定义上应面向未来可迁移到图形化界面。
- 推荐理由应尽量产品化，避免只输出原始字段，应翻译成用户能理解的决策语言。
- 成功标准不是单次跑通，而是持续输出可执行、可解释、可被用户采纳的方案。
- 产品不追求替代地图导航本身，而是专注解决“在哪里会合最合适”这一上游决策问题。

## Testing Decisions

- 好的测试应聚焦用户可观察的外部行为，而不是内部实现细节。例如，应验证在给定约束下是否返回可行方案、是否正确过滤不满足约束的候选点、是否在无解时给出合理反馈，而不是测试内部中间变量值。
- 需要重点测试的模块包括自然语言意图结构化、会合候选点生成与过滤、约束校验、方案排序、无解时的退让与重试策略、结果解释生成。
- 对规则引擎相关能力，应优先采用确定性输入和预期输出明确的测试方式，验证边界值、异常值和典型场景。
- 对意图解析能力，应重点覆盖表达多样性、地址歧义、缺省参数、跨城场景和偏好词汇解析。
- 对候选点与排序逻辑，应重点测试在多候选点下是否正确剔除无效方案，并按核心指标进行稳定排序。
- 对失败与重试逻辑，应测试超时、地图接口异常、候选点不足、约束过严等情况，确保系统能稳定降级或给出解释。
- 对结果表达层，应测试成功、有解但方案少、完全无解、输入信息不足等不同响应形态。
- 测试应尽可能复用已有的确定性规划能力，并在其上补充产品层面的验收标准。
- 对产品层验收而言，一个好的结果至少应满足：用户输入可被识别、约束被尊重、方案具备可执行性、推荐理由能帮助决策。

## Success Metrics And Optimization Loop

本产品需要同时具备产品指标和 Agent 评估指标，两者共同构成后续自我优化闭环。指标的目的不是只做业务汇报，而是要反向驱动意图解析、候选点生成、排序策略和结果表达的持续优化。

核心指标定义如下：

- 请求成功率：发起规划请求后，系统成功返回至少一个可执行会合方案的比例。该指标用于衡量系统是否稳定输出结果。
- 可行方案产出率：在所有请求中，返回满足硬约束的候选方案数量大于零的比例。该指标用于判断候选点生成和约束筛选是否合理。
- 约束命中率：返回方案中完全满足用户输入硬约束的比例。若出现任何超出用户设置上限的结果，则视为未命中。该指标用于监督规则层是否真正守住底线。
- 推荐采纳率：用户最终选择系统推荐方案的比例。若未来存在分享、确认、跳转导航或显式确认动作，可用这些行为作为采纳信号。
- 首推方案采纳率：用户是否直接采纳排序第一的方案。该指标用于判断排序模型与用户真实偏好的贴合程度。
- 无解率：系统在当前约束下返回无可行方案的比例。该指标用于发现默认候选点策略过窄、约束过严或地理编码问题。
- 放宽约束转化率：系统在提示放宽限制后，成功得到可行方案的比例。该指标用于判断“无解兜底”策略是否有效。
- 候选点利用率：自动生成的候选点中，最终进入可行方案集合的比例。该指标用于评估候选点生成质量，避免大量无效点位。
- 解释有效性指标：用户在看到结果后是否继续查看、分享、确认或重新要求解释。该指标用于判断结果文案是否足够帮助决策。
- 重规划成功率：在临时变更条件或一方迟到后，系统再次给出可执行方案的比例。该指标用于后续动态重规划能力评估。

建议进一步拆出 Agent 相关的过程指标：

- 意图解析准确率：系统对地址、角色、目的地、约束和偏好的解析正确率，可通过人工标注样本或回放集评估。
- 地址解析置信命中率：结构化地址与实际地理编码结果一致的比例，用于发现自然语言地址理解偏差。
- 约束抽取完整率：用户明确说出的限制条件中，被系统成功识别并进入规划输入的比例。
- 策略重试收益率：发生重试后，新增可行方案或提升方案质量的比例，用于评估策略节点是否真正产生价值。
- 排序一致性：同一输入在稳定地图数据下，多次运行后排序结果是否基本一致，用于避免结果漂移削弱用户信任。
- 质量评审拦截率：被评审模块识别为低质量方案并拦截的比例，用于验证 judge 层是否有效。

Agent 自我优化闭环建议定义为以下流程：

1. 采集每次请求的结构化输入、候选点、过滤原因、最终方案、用户后续行为和失败原因。
2. 将请求结果按成功、有解未采纳、无解、解释不足、地址歧义、重试后成功等标签分类。
3. 对高频失败类型做归因，区分是意图解析问题、候选点不足、规则过严、排序不符合用户偏好，还是结果解释不清楚。
4. 基于归因结果分别优化不同层：意图解析 prompt、候选点类型与搜索半径、评分权重、兜底放宽策略、结果文案模板。
5. 对优化前后版本做回放测试和离线对比，确认请求成功率、约束命中率和采纳率是否提升，再进入线上使用。

为支持上述闭环，单次请求至少应记录以下字段：

- 原始自然语言输入
- 解析后的结构化意图
- 地理编码结果及置信信息
- 自动生成的候选点列表
- 每个候选点的过滤原因
- 最终可行方案列表及排序分数
- 首推方案与备选方案的推荐理由
- 是否触发重试、放宽了哪些约束、重试后结果如何
- 用户最终是否选择某个方案、分享方案或发起重规划
- 最终失败原因或退出原因

第一阶段建议设定以下目标方向，而不是过早承诺绝对数值：

- 请求成功率持续提升，且明显高于人工手动试点的基线体验
- 约束命中率接近规则层理论上限，避免推荐越界方案
- 首推方案采纳率逐步提高，证明排序更贴近用户真实决策
- 无解率逐步下降，但不能以牺牲约束命中率为代价
- 重试收益率稳定为正，证明 Agent 编排确实优于单次静态规划

## Out of Scope

- 不覆盖通用单人导航与路线规划。
- 不在第一版支持三人及以上的多人复杂会合。
- 不在第一版支持多段多目的地联动规划。
- 不在第一版支持完整的实时路况持续追踪与秒级动态更新。
- 不在第一版支持完整社交协作能力，如多人投票、群组聊天室、共享实时位置。
- 不在第一版构建完整地图导航能力，汇合后的逐 turn 导航可交由现有地图服务承接。
- 不在第一版做长期个性化推荐系统或复杂偏好学习模型。
- 不在第一版优先解决海外多城市多语言场景。
- 不在第一版覆盖打车拼车计费、订单交易、支付或履约闭环。

## Further Notes

- 当前项目已经具备较强的技术 MVP 特征：自然语言输入、约束抽取、自动会合点规划、重试与评审机制、结果输出与过程追踪。下一阶段重点不是扩大技术范围，而是提升产品定位清晰度、结果解释力与决策闭环能力。
- 从产品角度看，最重要的不是生成一个点，而是帮助用户在有限时间内达成“公平且高效的会合决定”。
- 若后续进入图形化产品阶段，建议重点设计结果页，将其做成清晰的“方案决策卡片”，突出推荐理由、取舍差异和可分享性。
- 后续可以逐步增加出发时间建议、动态重规划、点位偏好控制和同行人确认机制，以增强实际使用价值。
- 产品指标不应只作为业务展示看板，还应成为 Agent 优化输入，持续反哺意图理解、候选点生成、排序策略和结果解释。
