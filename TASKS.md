# 结伴出行会合点规划产品开发任务清单

本文档将 [`PRD.md`](/Users/gethin/workspace/jiebanerxing/PRD.md) 拆解为可执行的研发任务，重点服务于当前 MVP 的产品化推进。优先级分为 `P0 / P1 / P2`：

- `P0`：构成最小可用闭环，优先完成
- `P1`：显著提升体验和可用性，建议在 P0 稳定后推进
- `P2`：中期增强项，暂不阻塞 MVP

## 当前阶段目标

当前阶段的核心目标不是扩大问题范围，而是把现有能力从“能规划”推进到“可解释、可评估、可迭代的产品 MVP”。

本阶段完成标准：

- 用户可以用自然语言发起一次结伴出游会合规划
- 系统能稳定返回至少 1 个满足硬约束的可执行方案
- 系统能给出清晰的推荐理由与备选方案差异
- 系统能记录关键指标，用于后续自优化

## 阶段结论

当前代码和文档状态已经基本达到 `P0`：

- `P0` 输入输出契约已通过 [`SPEC.md`](/Users/gethin/workspace/jiebanerxing/SPEC.md) 和结构化 `response_payload` 固化
- 自然语言意图解析、候选点生成、约束过滤与排序已经形成最小可用闭环
- 结果表达已升级为面向决策的结构，包含推荐理由、备选差异、无解建议、分享摘要和重规划差异
- 指标、日志、回放样本和基础测试已接入，可支撑后续 Agent 自优化

仍需持续优化但不阻塞 `P0` 判定的事项：

- 本地 `LM Studio` 可用性和端到端耗时仍受运行环境影响
- 真实用户行为数据尚少，当前“采纳率/分享率”骨架已具备，但还没有形成稳定业务数据
- 等待时间优化目前以“提示和建议出发时间”为主，尚未完全进入排序内核

下面的 `P0` 任务默认视为已完成，后续重点转入 `P1`。

## P0

### 1. 明确 P0 输入输出契约

目标：
- 统一“用户请求 -> 结构化意图 -> 规划结果 -> 用户响应”的最小产品契约

任务：
- 定义 P0 请求字段规范，包括出发地、目的地、角色、约束、偏好、候选点模式
- 定义 P0 响应字段规范，包括推荐方案、备选方案、核心指标、推荐理由、失败原因
- 明确硬约束字段与软偏好字段的边界
- 明确无解时的返回结构

验收标准：
- 同一请求结构可被 `intent`、`planner`、`response` 三层稳定消费
- 返回结果不依赖临时字段拼接
- 成功与失败场景都有一致的结构化输出

### 2. 强化自然语言意图解析

目标：
- 让用户可以稳定输入“谁从哪里出发、去哪里、有什么限制”

任务：
- 补齐地址、角色、目的地、约束和偏好的抽取规则
- 明确默认值策略和缺省参数回填规则
- 增强歧义输入处理，例如跨城、模糊地名、未明确角色
- 增加意图解析结果的校验与错误提示
- 为意图解析建立回放样本集

验收标准：
- 对典型双地出发场景可稳定提取结构化意图
- 用户明确说出的硬约束不会丢失
- 缺省情况下能使用保守默认值继续规划
- 地址歧义或字段缺失时能返回清晰提示，而不是静默失败

### 3. 稳定候选会合点生成

目标：
- 自动生成足够可用、现实中易执行的会合点

任务：
- 明确候选点类型优先级，如地铁站、商场、停车场、交通枢纽
- 梳理自动候选点生成的参数策略，包括半径、数量、关键词
- 记录候选点过滤原因，支持后续调试与优化
- 为跨城区与长距离目的地补充候选点策略
- 评估候选点不足时的扩圈和重试逻辑

验收标准：
- 常见周末出游场景下，系统能够生成足够候选点用于筛选
- 过滤后的结果能解释“为什么这个点不可用”
- 候选点不足时系统能自动扩展搜索，而不是直接失败

### 4. 固化硬约束过滤与排序逻辑

目标：
- 确保结果可信，先守住底线，再做优化

任务：
- 明确硬约束过滤顺序，包括乘客通勤时长、司机绕路时长、最大等待时长
- 梳理排序所依赖的核心指标及权重
- 把“均衡方案 / 最快方案 / 最少等待 / 最少绕路”定义成稳定输出策略
- 增加排序稳定性检查，避免同样输入结果频繁漂移
- 增加无解原因归类

验收标准：
- 所有返回方案都满足硬约束
- 结果至少包含 1 个主推荐方案和若干备选方案
- 排序结果与指标解释保持一致
- 无解时能说明是候选点不足、约束过严还是地理解析异常

### 5. 升级结果表达为“决策结果”

目标：
- 把当前偏技术的输出升级为用户能快速采纳的结果页语言

任务：
- 设计统一的结果结构，包括推荐标题、方案摘要、关键指标、推荐理由
- 为每个方案生成清晰的取舍说明
- 输出“为什么推荐这个方案，而不是另外两个”
- 无解时增加约束放宽建议
- 统一成功、有解但方案少、完全无解三类文案模板

验收标准：
- 用户能看懂每个方案的差异
- 推荐理由不是原始字段堆叠，而是面向决策的表达
- 无解场景下用户知道下一步应该调整什么

### 6. 建立指标与日志骨架

目标：
- 为后续 Agent 自优化建立最小数据闭环

任务：
- 定义核心埋点事件，如请求发起、意图解析成功、候选点生成、方案返回、无解、重试、用户采纳
- 定义关键指标字段，包括请求成功率、可行方案产出率、约束命中率、首推方案采纳率、无解率
- 扩展 trace 结构，补充过滤原因、重试收益、失败分类等信息
- 设计最小回放样本格式，用于离线评估
- 确保日志结构适合后续统计和分析

验收标准：
- 每次请求都能留下可分析的结构化记录
- 能从日志中计算出至少成功率、无解率、约束命中率
- 能区分意图问题、候选点问题、排序问题和外部接口问题

### 7. 建立 P0 验收样本集

目标：
- 不靠感觉判断是否完成 P0

任务：
- 收集 10 到 20 条真实场景样本，覆盖市内、跨区、跨城、严格约束、宽松约束、无解情况
- 为每条样本定义预期行为，不要求唯一答案，但要求满足基本边界
- 建立固定回放入口
- 为关键样本记录人工评估结论

验收标准：
- 新版本可稳定回放样本集
- 核心样本不出现明显回归
- 评估结果能辅助判断是否值得上线或继续迭代

## P1

### 8. 引入偏好化结果策略

目标：
- 让不同用户能看到更符合自己决策习惯的方案排序

任务：
- 支持“优先少等待”“优先少绕路”“优先快到达”等显式偏好
- 支持“少换乘”“更公平”等增强偏好，但初期权重低于主偏好
- 让偏好影响排序和推荐理由
- 区分“原始等待”和“可通过建议晚点出发优化后的等待”
- 记录用户偏好与最终采纳结果，用于后续调权

验收标准：
- 用户显式表达偏好时，排序结果能响应变化
- 推荐理由能体现偏好对结果的影响

### 8A. 支持复杂反馈与方案选择

目标：
- 让用户在第一轮看到方案后，不必重写完整请求，也能继续协商结果

任务：
- 定义 `preference_update / plan_feedback / option_selection` 三类二轮输入
- 支持第一版最关键的 5 类反馈：
  - 等待反馈：太久、可以晚点出发、希望少等
  - 绕路反馈：不想绕太远、我可以多开一点
  - 乘客体验反馈：少换乘、少步行、别太折腾
  - 点位反馈：不好停车、不好认、优先地铁站、不要商圈
  - 方案选择反馈：选第几个、确认某个会合点、沿用上一轮方案
- 明确每类反馈分别映射到：偏好更新、候选点过滤、约束更新、重规划动作或执行确认
- 为“选择方案”补充执行态输出，如分享摘要、后续重规划上下文

验收标准：
- 用户可以基于上一轮结果表达局部修正，而不必重新描述完整需求
- 系统能区分“继续优化”和“直接选定方案”
- 复杂反馈会稳定落到有限的系统动作上，而不是被当作全新请求从头处理

### 9. 增加分享与确认闭环

目标：
- 从“算完”走向“用起来”

任务：
- 设计可分享的方案摘要结构
- 支持生成便于转发给朋友的会合方案文本
- 记录是否发生分享、确认、二次讨论等行为

验收标准：
- 用户能低成本把方案发给同行人
- 分享后的内容足以让对方理解会合安排

### 10. 增加无解兜底体验

目标：
- 即使第一次无解，也尽量帮助用户走到可执行结果

任务：
- 为无解原因建立标准分类
- 增加“放宽哪项约束更有效”的建议逻辑
- 评估自动放宽与用户确认放宽两种策略

验收标准：
- 无解时不只是失败提示，而是提供明确下一步
- 放宽约束后成功得到方案的比例可被统计

### 11. 动态重规划能力

目标：
- 支持临出发前或途中条件变化

任务：
- 定义重规划触发事件，如迟到、堵车、地铁延误
- 设计在原方案基础上的增量重规划逻辑
- 记录重规划前后差异

验收标准：
- 条件变化后，系统可基于原请求快速重新给出可行方案
- 用户能看懂新旧方案差异

### 12. P1 框架能力迁移

目标：
- 把当前 P1 中“会话、反馈继续执行、恢复状态”的外层自建机制，逐步迁回 LangGraph 原生能力，避免继续重复造轮子

当前状态：
- `P1-A` 基本完成：`session_id == thread_id` 已对齐，图已使用本地持久化 checkpointer
- `P1-B` 基本完成：首轮规划后会进入等待反馈态，后续反馈通过 `Command(resume=...)` 恢复
- `P1-C` 基本完成：交互式 Python CLI 已接入，`run.sh` 已降级为 launcher
- `P1-D` 本轮完成主路径瘦身：`session_store` 保留产品态摘要和索引，不再承担图执行恢复

任务：
- `P1-A 框架会话对齐`
  - 让 `session_id == thread_id`
  - 所有 `app.invoke()` 明确带 `configurable.thread_id`
  - 图 compile 使用持久化 checkpointer，而不是仅 `InMemorySaver`
- `P1-B 协商中断点`
  - 在“输出方案后”等待反馈

- `P1-C CLI 交互重构`

- `P1-D 自建状态层瘦身`

  - 删除对 `latest_intent/latest_response` 的主恢复依赖
  - 保留会话索引、摘要、已选方案、反馈历史

验收标准：
- 同一 `session_id` 下的后续反馈通过 LangGraph `thread_id` 恢复，而不是通过文件手工回填
- 第一轮规划完成后，图会进入等待反馈态
- `run.sh` 不再承担业务编排逻辑，只作为 launcher
- `session_store` 保留产品态摘要，但不再承担图执行恢复

## 建议实施顺序

建议按以下顺序推进：

1. P0 输入输出契约
2. 自然语言意图解析
3. 候选点生成与过滤
4. 约束过滤与排序
5. 结果表达升级
6. 指标与日志骨架
7. P0 验收样本集

这个顺序的原因是：

- 先把接口和边界定住，后面改动成本最低
- 先确保“能稳定算对”，再优化“说得好不好”
- 先有日志和样本，后续 Agent 自优化才有依据

做到这一步后，项目就会从“技术原型”明显迈向“产品 MVP”。
